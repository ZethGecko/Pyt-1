# Plan: Módulo de Auditoría - Sistema de Gestión de Transporte

## 1. Objetivo

Implementar un módulo de auditoría centralizado que registre todas las operaciones **CREATE, UPDATE y DELETE** sobre las entidades del sistema, permitiendo consultar quién hizo qué, cuándo y sobre qué dato.

---

## 2. Análisis del Estado Actual

### 2.1 Auditoría existente (parcial)
| Tabla | Tipo |
|---|---|
| `historial_tramite` | ✅ Ya existe — registra cambios de estado/transferencia de trámites específicamente. Solo cubre movimientos de departamentos. |
| Otras tablas | ❌ No tienen ningún mecanismo de auditoría. |

### 2.2 Limpieza identificada
— `historial_tramite` se quedaría **intacta** porque registra lógica de negocio específica (transferencias entre departamentos), que es diferente a un log genérico de auditoría.

## 3. Arquitectura propuesta

### 3.1 Entidad de auditoría
**Tabla:** `audit_log`

| Campo | Tipo | Descripción |
|---|---|---|
| `id` | BIGSERIAL PK | Identificador único |
| `tabla_afectada` | VARCHAR(100) | Nombre de la tabla modificada |
| `registro_id` | BIGINT | ID del registro afectado |
| `accion` | VARCHAR(10) | `INSERT` / `UPDATE` / `DELETE` |
| `fecha_accion` | TIMESTAMP | Momento del cambio |
| `usuario_id` | BIGINT FK → `users(id_usuarios)` | Usuario que realizó la acción |
| `ip_address` | VARCHAR(45) | IP del cliente (si está disponible) |
| `datos_anteriores` | JSONB | Estado anterior del registro (solo UPDATE/DELETE) |
| `datos_nuevos` | JSONB | Estado nuevo del registro (solo INSERT/UPDATE) |
| `descripcion` | VARCHAR(255) | Descripción legible del cambio |

**Índices recomendados:**
- `ix_audit_log_tabla_registro` sobre `(tabla_afectada, registro_id)`
- `ix_audit_log_fecha` sobre `(fecha_accion DESC)`
- `ix_audit_log_usuario` sobre `(usuario_id)`

### 3.2 Estrategia de captura — Hibernate Envers

Se usará **Hibernate Envers** como mecanismo de captura automática porque:
- Es la opción con menor fricción en proyectos Spring Boot / JPA existentes (ya usados en el proyecto).
- Detecta automáticamente INSERT/UPDATE/DELETE sin modificar cada servicio.
- Genera revisiones con snapshots completos del estado de la entidad.
- Ya soporta PostgreSQL / JSONB.

**Enfoque alternativo a evaluar:** un `AuditorAware` propio (que es más flexible pero requiere más código) se puede usar como complemento para capturar el usuario desde el contexto de seguridad actual, algo que Envers no captura por defecto sin configuración adicional.

## 5. Diseño técnico

### 5.1 Flujo de captura del usuario

```java
// Componente que extrae el usuario del contexto de seguridad de Spring
@Component
public class AuditorAwareImpl implements AuditorAware<String> {
    @Override
    public Optional<String> getCurrentAuditor() {
        Authentication auth = SecurityContextHolder.getContext().getAuthentication();
        if (auth == null || !auth.isAuthenticated()) return Optional.of("SYSTEM");
        Object principal = auth.getPrincipal();
        if (principal instanceof UserDetails) {
            return Optional.of(((UserDetails) principal).getUsername());
        }
        return Optional.of(principal.toString());
    }
}
```

### 5.2 Configuración de Envers por entidad

Cada entidad a auditar se anota con `@Audited`. Las entidades auditadas también deben llevar `@RevisionEntity` para capturar la fecha de modificación y el usuario:

```java
@Entity
@Audited
@Table(name = "users")
public class Users { ... }
```

Se crea una entidad `AuditRevision` con:
- `id` (revision number)
- `timestamp` (fecha)
- `username` (quien realizó la acción) — capturado mediante `RevisionListener`

### 5.3 Estructura de tablas que Envers crea automáticamente

| Tabla Envers | Contenido |
|---|---|
| `revinfo` | `rev` (número de revisión), `revtstmp` (timestamp), `username` |
| `users_AUD` | Historial de cambios de `users` |
| `roles_AUD` | Historial de cambios de `roles` |
| `tramite_AUD` | Historial de cambios de `tramite` |
| ... | (una por cada tabla auditada) |

### 5.4 Consulta de registros de auditoría

Se crea un **servicio `AuditLogService`** que consulta datos de las tablas `_AUD` generadas por Envers y permite:
- Buscar por tabla y registro
- Buscar por rango de fechas
- Buscar por usuario
- Consultar el diff (valores anteriores → nuevos) de un cambio

## 6. Implementación — Fases

### FASE 1: Backend — Infraestructura de auditoría

**Pasos:**

1. **Agregar dependencia de Hibernate Envers** a `pom.xml`
2. **Crear `AuditRevision`** (entidad de revisión) con `@RevisionEntity(AuditRevisionListener.class)`
3. **Crear `AuditRevisionListener`** que captura `username` desde el contexto de seguridad
4. ** Anotar cada entidad** con `@Audited`
   - Orden sugerido: `Users` → `Roles` → `Departamento` → `Empresa` → `Gerente` → `PersonaNatural` → `Vehiculo` → `Tramite` → `Solicitud` → `DocumentoTramite` → `Inspeccion` → `FichaInspeccion` → `InspeccionInstancia` → `Publicacion` → continuar con el resto
5. **Configurar Envers** en `application.properties`:
   - `org.hibernate.envers.audit_table_suffix=_AUD`
   - `org.hibernate.envers.revision_field_name=rev`
   - `org.hibernate.envers.revision_on_collection_change=false` (optimización)

6. **Crear `AuditLogService`** con métodos de consulta sobre las tablas `_AUD`
7. **Crear `AuditLogDTO`** para estructurar las respuestas de la API

### FASE 2: API REST de auditoría

**Endpoint propuesto:**

```
GET  /api/audit                            → Lista paginada de todos los logs
GET  /api/audit/{tabla}/{id}               → Historial de cambios de un registro específico
GET  /api/audit/search                     → Búsqueda con filtros (tabla, usuario, fecha desde/hasta, acción)
GET  /api/audit/{tabla}/{id}/{revision}    → Estado de un registro en una revisión específica
```

**DTO de respuesta de auditoría (`AuditLogResponseDTO`):**

```json
{
  "tablaAfectada": "tramite",
  "registroId": 42,
  "accion": "UPDATE",
  "fechaAccion": "2026-05-20T10:30:00",
  "usuario": "admin",
  "descripcion": "Actualización del trámite 'TRAM-001': estado PENDIENTE → APROBADO",
  "datosAnteriores": { "estado": "PENDIENTE", "prioridad": "normal" },
  "datosNuevos": { "estado": "APROBADO", "prioridad": "alta" }
}
```

**Filtros soportados:**
- Por tabla
- Por registro ID
- Por acción (`INSERT` / `UPDATE` / `DELETE`)
- Por usuario
- Por rango de fechas (`fecha_desde`, `fecha_hasta`)

### FASE 3: Seguridad

Los endpoints de auditoría requieren rol `ADMIN` o `SUPER_ADMIN` porque exponen información sensible de quién modificó qué.

→ Agregar a `SecurityConfig.java`:
```java
.requestMatchers("/api/audit/**").hasAnyRole("ADMIN", "SUPER_ADMIN")
```

### FASE 4: Migración de datos (si existe producción)

Si ya hay datos en producción, la migración requiere crear las tablas `_AUD` con el esquema actual. Envers no soporta retro-auditoría nativa sobre datos existentes — se evalúa según el volumen de datos. Si el volumen es bajo, se puede poblar manualmente desde `historial_tramite` existente; si es alto, solo se audita desde el momento de la migración en adelante.

### FASE 5: Frontend — Panel de Auditoría

Vistas a implementar en el frontend:

| Vista | Descripción |
|---|---|
| **Listado general** | Tabla paginada con todos los eventos de auditoría, columnas: fecha, usuario, tabla, acción, descripción, ID registro |
| **Filtros laterales** | Por tabla, acción, usuario, rango de fechas |
| **Detalle de cambio** | Modal/diálogo mostrando `datos_anteriores` vs `datos_nuevos` con campos resaltados |
| **Historial por registro** | Timeline de todos los cambios de un registro específico (ej: trámite #42) |
| **Exportar** | Opción de exportar a CSV/Excel |

## 8. Consideraciones y riesgos

| Consideración | Detalle |
|---|---|
| **Volumen de tablas `_AUD`** | Cada tabla auditada duplica su estructura en disco. Monitorear crecimiento. Considerar particionado por fecha en `audit_log` o `revinfo` si crece mucho. |
| **Borrado de datos sensibles** | Envers no borra registros históricos al borrar el registro original — cumplimiento con normativas de retención. |
| **Rendimiento** | Envers agrega overhead en cada INSERT/UPDATE/DELETE. Evaluar en entornos de alta concurrencia. Se puede desactivar Envers por entidad si es necesario. |
| **Registro de IP del cliente** | Requiere configuración adicional en `AuditRevisionListener` para capturar `HttpServletRequest`. |
| **Campos muy grandes** | Para campos como `observaciones` (TEXT), JSONB puede almacenarlos sin problema pero aumenta el tamaño de las tablas `_AUD`. |
| **Retrocompatibilidad** | El frontend actual no tiene panel de auditoría — la Fase 5 requerirá nuevo trabajo de frontend. |
| **Valores nulos en JSONB** | Envers puede guardar valores `null` dentro de JSONB — se maneja en el DTO. |

## Preguntas para el usuario

1. **¿Qué método de captura prefieres?** Hibernate Envers (rápido, menos código) o un listener/Interceptor manual (más control, más código)?
2. **¿Debe capturarse también la IP del cliente?** Requiere acceso a `HttpServletRequest` en el listener de auditoría.
3. **¿Hay una política de retención de logs de auditoría?** (ej: borrar registros mayores a 2 años, o nunca borrar)
4. **¿El frontend es Angular?** ¿Qué framework/stack usa el frontend para planificar la Fase 5? (Veo `localhost:4200` y `localhost:3000` en CORS — probablemente Angular o React)
5. **¿Debe incluirse en el panel de auditoría un `diff` visual (comparador de valores anteriores vs nuevos)?**
